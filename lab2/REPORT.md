## 1. Теоретическая часть

### 1.1 Постановка задачи

Требуется система, позволяющая:

- **вставлять** географически расположенные объекты с координатами (Lat, Lng) и идентификатором;
- **искать** объекты в пределах заданного радиуса $r$ (км) от координаты запроса - не точное совпадение, а приближённый поиск.

Поиск ближайших объектов на поверхности сферы - задача **геопространственного** индексирования, для которой существуют два основных подхода: **geohash**-индекс и **K-D дерево**.

### 1.2 Формула Haversine

Для вычисления сферического расстояния между двумя точками $(\phi_1, \lambda_1)$ и $(\phi_2, \lambda_2)$:

$$d = 2R \arcsin\sqrt{\sin^2\!\left(\frac{\phi_2 - \phi_1}{2}\right) + \cos\phi_1\cos\phi_2\sin^2\!\left(\frac{\lambda_2 - \lambda_1}{2}\right)}$$

где $R = 6371$ км - радиус Земли. Формула даёт корректные результаты вплоть до антиподных точек.

### 1.3 Geohash-индекс

**Geohash** кодирует координаты (Lat, Lng) в строку base-32 с настраиваемой точностью (precision). Бит за битом координаты перемежаются (interleave), что обеспечивает **пространственную локальность**: схожие строки ↔ близкие точки.

Ячейка точности $p$ имеет приблизительный размер:

| precision | ширина × высота |
|:---------:|:---------------:|
| 3 | ~156 × 156 км |
| 4 | ~39 × 20 км |
| 5 | ~4.9 × 4.9 км |
| 6 | ~1.2 × 0.6 км |

Алгоритм поиска в радиусе $r$:

1. Кодируем центр запроса с precision $p$.
2. BFS-расширением находим все соседние ячейки, центры которых находятся не дальше $r + d_\text{cell}$, где $d_\text{cell}$ - диагональ ячейки.
3. Из найденных ячеек извлекаем кандидатов и фильтруем по точному расстоянию (Haversine).

Ключевое свойство: при правильно подобранном precision большинство кандидатов уже находятся в нужном радиусе, а число лишних ячеек минимально.

### 1.4 K-D дерево

**K-D дерево** - бинарное дерево разбиения пространства, чередующее разрезающие гиперплоскости по каждому измерению. Для геопоиска используется 2D k-d дерево с координатами (Lat, Lng).

Два режима построения:

- **Online (Insert)**: новые точки добавляются поочерёдно в существующее дерево. Дерево может стать несбалансированным.
- **Balanced (BuildBalanced)**: все точки сортируются по медиане на каждом уровне, что даёт сбалансированное дерево высотой $\lceil \log_2 N \rceil$.

Поиск ближайших в радиусе - обход дерева с отсечением (pruning): поддерево отсекается, если ближайшая точка в его регионе дальше $r$.

Ожидаемая сложность поиска:
- Balanced: $O(\sqrt{N} + k)$, где $k$ - число найденных точек.
- Online: $O(N)$ в худшем случае для несбалансированного дерева.

### 1.5 Brute Force (baseline)

Линейный перебор всех $N$ точек с вычислением расстояния Haversine: сложность $O(N)$ на запрос. Используется как **эталон** корректности при тестировании.

## 2. Практическая часть

### 2.1 Общий интерфейс Searcher

Код: [`internal/geoindex/geoindex.go`](internal/geoindex/geoindex.go)

```go
type Point struct {
    ID  string
    Lat float64
    Lng float64
}

type Result struct {
    Point    Point
    Distance float64 // км
}

type Searcher interface {
    Insert(p Point)
    FindNearby(lat, lng, radiusKm float64) []Result
    FindKNearest(lat, lng float64, k int) []Result
    Count() int
}
```

Результаты `FindNearby` отсортированы по расстоянию (ближайшее первым).

### 2.2 Реализация Geohash-индекса

Код: [`internal/geoindex/geoindex.go`](internal/geoindex/geoindex.go), [`internal/geo/geohash.go`](internal/geo/geohash.go)

```go
func New(precision int) *Index
func (idx *Index) Insert(p Point)
func (idx *Index) FindNearby(lat, lng, radiusKm float64) []Result
```

Внутреннее хранилище: `map[string][]Point` - ключ geohash ↔ список точек.

Функция `candidateCells` реализует BFS-расширение от центральной ячейки: начиная с центра, добавляет соседей (`Neighbors(hash)`) до тех пор, пока расстояние от центра запроса до центра новой ячейки не превышает `radiusKm + cellDiagonal`. Это гарантирует покрытие всего круга запроса без избыточного перебора.

### 2.3 Реализация K-D дерева

Код: [`internal/kdtree/kdtree.go`](internal/kdtree/kdtree.go)

```go
func New() *KDTree
func (t *KDTree) Insert(p geoindex.Point)
func (t *KDTree) BuildBalanced(points []geoindex.Point)
func (t *KDTree) FindNearby(lat, lng, radiusKm float64) []geoindex.Result
func (t *KDTree) FindKNearest(lat, lng float64, k int) []geoindex.Result
```

- Чередование осей: чётные уровни - по Lat, нечётные - по Lng.
- `BuildBalanced` рекурсивно выбирает медиану по текущей оси.
- `FindKNearest` использует max-heap для хранения $k$ ближайших точек.
- Расстояние при обходе вычисляется через Haversine.

### 2.4 Brute-force сканер

Код: [`internal/brute/brute.go`](internal/brute/brute.go)

```go
func New() *Scanner
func (s *Scanner) Insert(p geoindex.Point)
func (s *Scanner) FindNearby(lat, lng, radiusKm float64) []geoindex.Result
```

Хранит все точки в срезе. `FindNearby` итерирует по всем точкам и отфильтровывает те, что дальше `radiusKm`. Результаты сортируются по расстоянию.

### 3.2 Методика замеров

Использован единый `runBatchBenchmark` с `b.Loop()`:

- **Insert**: каждая итерация вставляет $N$ случайно перемешанных точек в **новый** индекс; время на точку замеряется точно.
- **FindNearby**: индекс строится **один раз** вне таймера; каждая итерация - один вызов `FindNearby`. Радиус запроса - 10 км, центр - Москва (55.75°N, 37.62°E).
- **BuildKDTree**: каждая итерация строит новое дерево из $N$ перемешанных точек.

Для каждой серии вычисляется **95% доверительный интервал**:

$$\text{CI}_{95} = 1.96 \cdot \frac{s}{\sqrt{n}}$$

Тестовые данные - кластеризованные точки в нескольких регионах мира (функция `gen.RandomRegionPoint`), что соответствует реальному распределению POI.

## 4. Вывод

Реализованы три алгоритма геопространственного поиска с общим интерфейсом `Searcher`:

1. **Geohash-индекс** (precision 5 и 6) - хорошая пропускная способность при вставке; поиск медленнее K-D дерева, но намного быстрее brute-force при больших N. Оптимален для онлайн-вставок при умеренных требованиях к скорости поиска. Precision 5 предпочтительнее precision 6 для радиусов 5–50 км.

2. **K-D дерево** - наилучшее время поиска ($O(\sqrt{N})$). Balanced-вариант даёт минимальную задержку запроса ценой дорогого построения; online-вариант - разумный компромисс для потоковых вставок.

3. **Brute Force** - корректный эталон, $O(N)$ на запрос. Приемлем лишь при N < 10 000.

Для всех замеров вычислены 95% доверительные интервалы. Тесты на корректность включают проверку через brute-force оракул с допуском на граничные эффекты geohash, а также фаззинг с произвольными координатами и радиусами.

**Рекомендации по применению:**

| Сценарий | Рекомендация |
|:---------|:-------------|
| N < 5 000, редкие вставки | brute-force (простота) |
| N < 100 000, смешанный R/W | geohash-p5 |
| N > 100 000, много запросов | kd-tree balanced (периодическая перестройка) |
| Поток вставок + запросы | kd-tree online |
